## Understanding form validation

Chassis CSS provides flexible options for implementing form validation:

- **Native validation**: Uses the browser's built-in `:invalid` and `:valid` pseudo-classes to style form controls based on validation state.
- **Progressive enhancement**: Chassis uses the `.was-validated` class on forms to control when validation styles appear. This prevents invalid styles from showing when the page first loads.
- **Server-side fallback**: The `.is-invalid` and `.is-valid` classes provide direct control for server-side validation scenarios.
- **Custom validation messages**: Add contextual feedback to users with `.valid-feedback` and `.invalid-feedback` elements.
- **Opt-in icons**: Add `.validation-icons` to a wrapper (e.g. the `<form>`) to render success/error glyphs inside every supported input it contains. Without it, only the colors, border, and shadow change.
- **API integration**: All validation styles work with the browser's [Constraint Validation API](https://html.spec.whatwg.org/multipage/form-input-infrastructure.html#the-constraint-validation-api), allowing you to create custom validation logic with JavaScript.
- **Accessible feedback**: Position validation messages consistently outside input groups for optimal screen reader compatibility.

## Client-side validation

For custom Chassis validation styles, add the `novalidate` attribute to your `<form>` element. This disables browser default tooltips but still allows JavaScript validation using the Constraint Validation API.

Key features of Chassis validation:

- Custom feedback styles with contextual colors and icons
- Support for all form controls including input groups
- Consistent positioning of feedback messages
- JavaScript integration for dynamic validation

## Server-side validation

For applications that validate on the server, Chassis CSS provides the `.is-valid` and `.is-invalid` state classes that can be applied to form controls directly, without needing the `.was-validated` parent class.

<Callout type="warning" title="Accessibility best practices">
  For invalid fields, always connect error messages to their inputs using `aria-describedby` to ensure screen readers announce feedback properly. Multiple IDs can be referenced if a field has both help text and error messages.
</Callout>

## Validation icons

Validation icons are opt-in at the **wrapper level**. Add the `.validation-icons` class to any ancestor — typically the `<form>`, but it also works on `.form-field`, `.input-group`, or even a single `.form-input` — and every supported descendant input renders the success or error glyph when it enters a validation state. Without an ancestor with this class, inputs still pick up validation colors, border, and shadow — only the icon is omitted.

Icons appear on text inputs, single-value `<select>` elements, color inputs, `.ghost-input` (used by `.chip-input` and the `.input-help` wrapper), and `.combobox-value`. Textareas, checkboxes, radios, switches, file, and range inputs do not receive icons by design.

## Persistent Form help

The `.form-help` component is hidden by default when validation feedback is present so the active message takes precedence. Add the `.always-show` class to keep help text visible alongside validation feedback. Always place form help and validation feedback elements as siblings after the input or input group.

## Accessibility

Validation feedback only works for every user when it's wired up with the right ARIA attributes. The example below demonstrates the recommended pattern for dynamic, JavaScript-driven validation — connecting the input to its feedback with `aria-describedby`, announcing changes through a live region, and toggling `aria-invalid` in sync with `.is-invalid`.

<Callout type="info" title="Accessibility checklist">
  Because `.invalid-feedback` and `.valid-feedback` are hidden with `display: none` when inactive, you can safely list all relevant IDs in a single static `aria-describedby` — screen readers skip `display: none` elements, so only the currently visible description is announced on focus. Add `aria-live="polite"` to feedback elements so errors injected via JavaScript are announced immediately without requiring the user to re-focus the field. Always set `aria-invalid="true"` on the input when it is invalid, and remove it when the error is cleared.
</Callout>

<Example code={`<div class="input-group">
    <span class="input-addon">@</span>
    <input type="text" class="form-input" id="username-input"
      aria-describedby="username-feedback username-help" placeholder="Username" required>
    </div>
    <div class="invalid-feedback" id="username-feedback" aria-live="polite"></div>
    <div class="form-help always-show" id="username-help">Enter your username</div>
    <div class="mt-medium">
      <button class="button primary" type="button" onclick="validateUsername()">Validate username</button>
  </div>`} />

<Code lang="js" code={`function validateUsername() {
    const input = document.getElementById("username-input")
    const feedback = document.getElementById("username-feedback")
    const invalid = !input.validity.valid
